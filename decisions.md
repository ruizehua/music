## ADR 001: 数据库选择

### 日期
2026-05-21

### 上下文
需要选择适合个人音乐应用的数据库，用于存储音乐元数据、歌单、用户信息等。

### 考虑的选项
1. **SQLite**
2. **MySQL / PostgreSQL**
3. **NoSQL (MongoDB)**

### 决策
选择 **SQLite**。

### 理由
- **轻量级**：无需独立数据库服务
- **部署简单**：单文件，易于备份
- **足够性能**：个人使用场景下性能充足
- **功能完整**：支持事务、索引等
- **Spring Data 支持**：与 Spring Boot 集成良好

### 后果
- 单机存储，不适合分布式场景
- 但对于个人使用场景，完全够用

## ADR 003: 音乐存储方式

### 日期
2026-05-21

### 上下文
需要选择音乐文件的存储方式。

### 考虑的选项
1. **本地文件系统 + SQLite (仅存元数据)**
2. **对象存储 (MinIO, etc.)**
3. **数据库存储 BLOB**

### 决策
选择 **本地文件系统 + SQLite**。

### 理由
- **简单直接**：易于管理和备份
- **性能好**：直接读取文件，无需转译
- **灵活**：音乐文件可以直接在文件系统中操作
- **避免数据库膨胀**：文件不存入数据库，保持轻量

## ADR 004: 管理后台技术栈

### 日期
2026-05-21

### 上下文
需要选择管理后台的前端技术栈。

### 考虑的选项
1. **Vue 3 + Vite + Element Plus**
2. **React + Ant Design**
3. **纯后端渲染 (JSP/Thymeleaf)**

### 决策
选择 **Vue 3 + Vite + Element Plus**。

### 理由
- **轻量快速**：Vite 构建速度快
- **组件库成熟**：Element Plus 组件丰富
- **类型安全**：支持 TypeScript
- **学习曲线低**：Vue 上手快

## ADR 005: 音乐播放库

### 日期
2026-05-21

### 上下文
需要选择 Android 端的音乐播放库。

### 考虑的选项
1. **ExoPlayer / Media3**
2. **系统 MediaPlayer**
3. **第三方库 (ExoPlayer 前的其他库)**

### 决策
选择 **Media3 (ExoPlayer)**。

### 理由
- **官方推荐**：Google 官方媒体库
- **功能强大**：支持流媒体、缓存、DASH/HLS
- **高度可定制**
- **活跃维护**
