## 项目特定说明

### 工作模式

#### 1. 主动模式
当用户提出任务时，Claude Code 应该：
1. **理解需求**：先分析任务是否清晰，必要时提问澄清
2. **制定计划**：创建 Todo 列表分解任务
3. **执行变更**：按照 `AGENTS.md` 中的规范进行开发
4. **验证结果**：运行测试确保功能正确
5. **更新文档**：如有必要，更新相关文档

#### 2. 解释模式
当用户要求解释代码时：
- 重点说明代码的功能和设计意图
- 解释关键实现细节
- 提供使用示例（如适用）
- 使用中文回复

#### 3. 审查模式
当用户要求代码审查时：
- 检查代码是否符合 `AGENTS.md` 中的规范
- 识别潜在的性能问题
- 检查安全风险
- 提供改进建议

### 任务处理流程

```
用户请求 → 理解确认 → 计划分解 → 执行开发 → 验证测试 → 完成反馈
    ↓
  如有疑问 → 主动询问 → 确认后继续
```

### 特殊场景处理

#### 大规模变更
当需要修改超过 10 个文件时：
1. 先列出所有变更计划
2. 征求用户确认
3. 分批次进行变更
4. 每批次完成后验证

#### 破坏性变更
涉及以下内容的变更必须先确认：
- 修改数据库表结构
- 删除核心功能模块
- 修改 API 接口
- 影响现有用户数据

#### 紧急修复
当需要快速修复生产问题时：
1. 优先保证修复的正确性
2. 简化方案，避免引入新问题
3. 事后补充测试和文档

## 快速开始指南

### 开发环境准备

#### 1. 克隆项目
```bash
git clone <repository-url>
cd music
```

#### 2. 后端开发环境
```bash
# 进入后端目录
cd backend

# 使用 Maven 构建
mvn clean install

# 运行应用
mvn spring-boot:run

# 运行测试
mvn test
```

#### 3. Android 开发环境
```bash
# 进入 Android 目录
cd android

# 使用 Gradle 构建
./gradlew assembleDebug

# 安装到设备
./gradlew installDebug

# 运行单元测试
./gradlew test

# 运行 UI 测试
./gradlew connectedAndroidTest
```

### 常用命令

#### Git 操作
```bash
# 创建功能分支
git checkout -b feat/music-player

# 提交代码（遵循 Conventional Commits）
git commit -m "feat(music): 添加音乐播放功能"

# 获取最新代码
git pull origin main

# 推送代码
git push origin feat/music-player
```

#### 代码检查
```bash
# 后端代码检查
cd backend
mvn checkstyle:check

# Android 代码检查
cd android
./gradlew lint
```

## 附录：常见问题

### Q: 如何处理陌生的代码？
**A**: 
1. 先查找相关的测试文件理解预期行为
2. 查看接口定义和文档
3. 追踪调用链路理解上下文
4. 如仍不清楚，主动询问团队成员

### Q: 如何处理遗留代码？
**A**:
1. 保持谨慎，避免不必要的改动
2. 确保有足够的测试覆盖
3. 改动前先与原作者或团队确认
4. 遵循逐步演进而非大规模重写原则

### Q: 如何处理技术债务？
**A**:
1. 在代码中添加 TODO 注释标注
2. 创建 Issue 记录债务内容
3. 在日常开发中逐步偿还
4. 重大技术债务需要专项处理计划
